# 001-true-streaming - Work Plan

## TL;DR (For humans)
<!-- Plain English for a non-engineer: NO file paths, NO todo numbers, NO wave/agent/tool names. -->

**What you'll get:** Streaming responses (`stream: true`) will actually stream — text and reasoning appear on the wire as the AI agent produces them, instead of the caller waiting in silence for the whole answer and then seeing it "replayed" all at once. This applies to both the Responses API and the Chat Completions API. Non-streaming requests keep working exactly as they do today.

**Why this approach:** Most of the plumbing needed already exists in the codebase but was never wired together — this plan connects the existing real-streaming capability to the actual HTTP endpoints. Text chunks are forwarded exactly as the agent produces them (no artificial "typewriter" delay added), which is the simplest approach and matches what "real-time" should mean.

**What it will NOT do:** It will not add a toggle/flag to switch between old and new behavior — the new streaming replaces the old one outright. It will not add usage/token tracking or the ability for the agent to actually execute tools (those are separate, already-tracked future items). It will not add any request-rate limiting.

**Effort:** Large
**Risk:** Medium - real-time streaming introduces new failure modes (a crashed agent mid-answer, a caller who disconnects early, a slow reader) that the current simpler design doesn't have to handle; this plan explicitly requires cleanup to happen reliably (releasing resources within 5 seconds) in all of those cases.
**Decisions to sanity-check:** (1) chunks are forwarded exactly as the agent sends them, with no smoothing/re-splitting for a nicer typing effect — you approved this already; (2) the automated tests prove correctness for 10 simultaneous streaming requests, not the full 50 mentioned in the original request — a bigger real-world load test would need to be run separately if that scale matters in practice.

## Scope
### Must have
- `/v1/responses` and `/v1/chat/completions` deliver `stream: true` content incrementally as the ACP agent produces it (FR-001), using the already-existing `AcpClient::send_prompt_streaming()`.
- Existing streaming event structure/semantics preserved for both APIs (event types, role/finish_reason/usage framing, reasoning vs. message distinction) — FR-002.
- Clean, well-formed completion signal on normal turn end, matching today's shape, including the zero-content edge case (FR-003, US2).
- Identifiable error/termination signal on agent error/disconnect mid-turn, no indefinite hangs (FR-004).
- Client disconnect stops further forwarding and releases resources (FR-005, US3).
- Unrecognized ACP update types (e.g. `available_commands_update`) keep being silently dropped at debug level (FR-006) — no behavior change here.
- Multiple concurrent streaming requests remain isolated, no cross-talk (FR-007, SC-005).
- Non-streaming requests behave exactly as today, unaffected (FR-008).
- Bounded buffering — memory does not grow unbounded for a slow-consuming client (FR-009, SC-006).
- Mock agent supports a configurable per-chunk delay so progressive delivery (SC-001, SC-002) is testable.

### Must NOT have (guardrails, anti-slop, scope boundaries)
- No feature flag / opt-in toggle — true streaming replaces the existing batch-then-replay streaming path outright.
- No changes to non-streaming (`stream: false`) request handling or its existing tests (`translate.rs:983-1055`).
- No new ACP update types recognized beyond `agent_thought_chunk`, `agent_message_chunk`, `tool_call`.
- No token usage tracking implementation — usage stays hard-coded per existing roadmap item (separate scope).
- No tool execution loop implementation (separate roadmap item).
- No artificial word re-splitting of raw agent chunks — forward each raw ACP chunk verbatim as one delta (Option A, user-approved).
- No admission-control / concurrency cap / rate limiting on streaming requests.
- No modification of `process_update()`'s existing match-arm logic for Responses API (reuse as-is); only its call site changes from batch to live.

## Commit strategy

- Wave boundaries are natural commit checkpoints: Wave 1 (Todos 1-5) lands the foundation (mock delay/crash/echo, `AcpStreamItem`-aware streaming task, chat translator, SSE helper, unit tests) without touching route handlers — safe to land independently. - No squashing across waves — preserve the granular history so a future `git bisect` or revert can isolate the streaming-task wrapper from the route-handler wiring from the test additions.
